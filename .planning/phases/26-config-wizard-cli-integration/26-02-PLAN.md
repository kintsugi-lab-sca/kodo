## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator shell → kodo polling CLI | argv y env vars no confiables |
| ~/.kodo/polling.pid → readPidFile | Otro proceso podría escribir un payload malicioso |
| spawn child detached → PID OS-level | El hijo corre desligado; el PID puede ser reusado tras crash |
| SIGTERM/SIGINT → cleanup handler | Race entre signal y `removePidFile` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26-02 | Tampering | PID file `~/.kodo/polling.pid` | mitigate | `chmod 0o600` PRE-rename en `writePidFile` (Security V14 — token-adjacent metadata; only owner reads/writes). `readPidFile` defensive shape check (`typeof pid === 'number'`, `typeof started_at === 'string'` → null) previene PID injection. Test source-hygiene: `grep -c "GITHUB_TOKEN\|api_key\|token" src/cli/polling-daemon.js === 0`. |
| T-26-03 | DoS (false positive) | runPollingStatusCli isPidAlive check | accept | Tras crash, el OS puede reciclar el PID. `kodo polling status` podría reportar `running` a un PID ajeno (e.g. vim). Probabilidad real en Mac/Linux dev box trivial (PID space 32-bit, segundos entre crash y rearranque). Aceptar como known limitation per RESEARCH Pitfall #1; documentar en summary. Reusar `isPidAlive` de `src/gsd/lock.js:67` sin enrichment. |
| T-26-04 | Availability | runPollingStartCli daemon path | mitigate | `child.unref()` INMEDIATO post-spawn (Pitfall #2 — sin esto el padre cuelga). `detached: true` + `stdio: 'ignore'` desliga al hijo del process group del padre. Si el padre crashea durante setup, el hijo lee config + arranca solo (sin rely on parent). Test integration: `start --no-daemon` + SIGINT en padre, assert hijo (mismo proceso en --no-daemon) exit 0 clean. |
