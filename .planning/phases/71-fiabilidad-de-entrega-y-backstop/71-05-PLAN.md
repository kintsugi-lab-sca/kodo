<objective>
Cerrar el GAP 2 / DELIV-04: el backstop mecánico de «In Review» CIERRA automáticamente issues de GitHub en vez de mandarlos a revisión. La premisa de diseño (D-13: «GitHub no implementa transición de estado igual que Plane → el provider degrada a no-op») es empíricamente FALSA: GitHub SÍ implementa `getTaskState`/`updateTaskState`/`addComment`, así que el capability-gate PASA. Combinado con el default `states.review:'closed'` de GitHub (`src/config.js:333`) y el passthrough hard de `updateTaskState` (que llama `client.updateIssue(..., {state:'closed'})`), el backstop CIERRA el issue en cualquier `SessionEnd` limpio — más destructivo que el «falso In Review» que D-12 asumió barato.

Fix LOCKED por el operador (2026-07-07): **no cerrar nunca** — gate de estado no-terminal. El backstop solo transiciona cuando el `reviewState` resuelto NO es un estado terminal/de cierre. Para GitHub (`'closed'`) queda no-op; para Plane (`'In review'`, no-terminal) procede como hoy. NO se adopta la alternativa del gate `verdict.action === 'pass'`.

Además, corregir la documentación que sostiene la premisa falsa (D-13 en `71-CONTEXT.md`, y las secciones equivalentes de `71-RESEARCH.md`): el no-op de GitHub pasa a deberse al gate de estado no-terminal, NO a la ausencia de capacidades.

Purpose: cierra la causa raíz T5 sin efecto secundario destructivo — la tarea nunca queda colgada en «In Progress» y el backstop NUNCA cierra un issue de GitHub que solo estaba en curso.

Output: un guard de estado terminal en `runReviewBackstop` (provider-agnostic vía `states.done` + reconocimiento de `'closed'`); tests que reproducen el escenario REAL de GitHub (con las 3 capacidades) y el de Plane; corrección de la premisa falsa en los dos `.md`.
</objective>
