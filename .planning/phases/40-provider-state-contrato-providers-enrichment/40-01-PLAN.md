<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| provider API → adapter | Plane state `name`/`group` and GitHub label `name` are untrusted, provider-controlled strings crossing into kodo's normalization layer |
| session state → provider client | `task_id`/`task_ref`/`project_id` from the session record are used to address an outbound HTTP fetch (getWorkItem / getIssue) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-01 | Denial of Service | mapPlaneState / mapGithubLabels over provider strings | mitigate | Map ONLY via `String.includes` case-insensitive; NEVER `RegExp`/`.match`/`.test` over provider input (D-10/D-11). Anti-ReDoS asserted by grep gate in Tasks 1-2. |
| T-40-02 | Information disclosure | normalized return value | mitigate | Output is a closed vocabulary of 5 literals; the raw provider state name / label is NEVER returned or echoed. Asserted by the contract-matrix vocabulary check (Task 3). |
| T-40-03 | Tampering / SSRF | getWorkItem / getIssue outbound fetch | accept | task ids/refs derive from the existing session record and flow through the SAME provider client boundary already used by getTask/listComments (no new URL surface, no new id source). No new mitigation beyond the existing client. |
| T-40-SC | Tampering | npm/pip/cargo installs | accept | This plan adds ZERO dependencies (no install tasks). No package-legitimacy gate required. |
</threat_model>
