<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| stdin → buildGsdContext | El hook lee `session` desde `state.json` (controlado por kodo) y `opts.brief` desde `main()` (derivado de `buildBriefFromTask` en Phase 9). El campo `session.summary` proviene del title/summary del work item en Plane (input externo). |
| stdout → Claude Code | El context block emitido por el hook se inyecta en el agente. Una inyección maliciosa en `summary` podría intentar escapar el comando slash. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Tampering | `safeTitle` escape de comillas | mitigate | `session.summary.replace(/"/g, "'")` defangs comilla-doble. Si un title viene con `"; rm -rf /; echo "`, queda como `'; rm -rf /; echo '` — no escapa la comilla envolvente. La inyección requeriría además newline (no se filtra) Y que Claude Code interprete el slash command parser inconsistentemente. Severidad baja: el hook ejecuta en una sandbox local, no como root, y el slash command `/gsd-quick` lo interpreta el agente, no el shell. |
| T-12-02 | Information Disclosure | `opts.brief` rendering | accept | El brief lo escribe `buildBriefFromTask` (Phase 9) a partir del task description. Si el description tiene secrets (API keys, tokens), aparecerán en el contexto del agente. Status quo Phase 9 — Phase 12 no introduce este vector. Mitigación corresponde al pipeline de ingest, no al hook. |
| T-12-03 | Denial of Service | Title arbitrariamente largo | accept | `session.summary` ya está limitado por Plane (~255 chars típicos). No hay loop sobre el título; replace es O(n) lineal. Severidad baja. |
</threat_model>
