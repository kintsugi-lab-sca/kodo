<objective>
Promote the kodo dashboard TUI from read-only to read-write: add the `d` (dismiss) capability that arms an inline `mode:'confirm'` prompt over a dead row, dispatches `DELETE /sessions/{id}` through a never-throws client on a second `d`, and maps the structured result to a transient footer — the **first conscious break of the v0.9 "TUI read-only" invariant** (backlog 999.1).

This plan delivers the **TUI half** of Phase 42 (DISMISS-02, DISMISS-03, the TUI layer of DISMISS-04). It consumes the HTTP contract `{ok, removed, actions:[{type,result}]}` / `409 {ok:false,error:'alive'}` defined by Plan 01, but shares ZERO files with it — both plans are wave 1.

Purpose: The dismiss must never crash React (every failure collapses to `{ok:false,error}` in `client.js`, mirroring `fetchComments`), must never dismiss a live session (inverse of the Enter guard), and must surface a preserved `.dirty` worktree honestly in the footer (lesson v0.9 37/38). The TUI guard is UX-only — the authoritative TOCTOU re-check lives server-side (Plan 01).

Output: `dismissSession` in `client.js`, a pure `mapDismissResult` in `select.js`, `mode:'confirm'` + the `d` handler + DISMISS_* exported constants + the unified transient footer in `App.js`/`SessionTable.js`, and tests covering the state machine, the 3-layer TUI guard, the never-throws client, and the footer mapping.
</objective>
