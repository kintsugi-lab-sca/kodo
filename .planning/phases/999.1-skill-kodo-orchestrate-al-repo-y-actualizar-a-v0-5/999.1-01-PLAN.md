    ### 4.2 Cuándo correr `kodo gsd verify`

    Reglas operativas:
    - Sesiones `[GSD phase N]` o `[GSD bootstrap]` que entran a Review →
      `kodo gsd verify <session-id>`. El CLI postea el comentario y transiciona
      el work item. Exit codes deterministas:
      - `0` pass — el CLI movió la tarea a Review, continúa con la ronda
      - `1` soft-fail — el CLI comentó el motivo, espera corrección humana
      - `2` hard-fail — same
      - `64..78` usage/not-found — error operacional, revisar el comando
    - Sesiones `[GSD quick]` → **NO ejecutes `kodo gsd verify`**. El CLI no las
      soporta (one-shot, sin `VERIFICATION.md`). Revísalas manualmente como
      cualquier sesión no-GSD: lee el comentario del agente y decide.

    ### 4.3 Stop nudge

    Cuando una sesión termina, el hook Stop envía un nudge al orchestrator. El
    texto varía por modo (cita literal del que tu sesión recibe):
    - Full: "Es una sesión GSD (fase N). Ejecuta `kodo gsd verify <session-id>`…"
    - Quick: "Es una sesión GSD quick (one-shot, sin VERIFICATION.md). Revísala
      manualmente…"
    - No-GSD: "Revisa el resultado y decide si pasa a Done o necesita más trabajo."

    Cuando recibas un nudge → ejecuta una ronda inmediatamente.

    ## SECCIÓN 5: Diagnóstico (D-10)

    Cuatro subsecciones síntoma → comando, en este orden:

    ### 5.1 Sesión stuck (>30min idle)

    1. `cmux read-screen --workspace <ref> --lines 30` para confirmar que no
       progresa.
    2. `kodo logs --follow --session-of <task-id>` para ver actividad reciente del
       NDJSON. El resolver es two-step (Phase 7 LOG-11): busca primero en
       `~/.kodo/state.json` por `task_id`, si no aparece hace head-line scan de
       los archivos en `~/.kodo/logs/`.
    3. Si no hay actividad reciente: `cmux send --workspace <ref> "<nudge>"` o
       escala según contexto.

    ### 5.2 Lock no se libera tras Stop

    1. `ls ~/.kodo/locks/` para ver locks per-repo activos. El nombre es el
       realpath del repo (Phase 8 GSD-10).
    2. `cat ~/.kodo/locks/<repo-realpath>.lock` — JSON con `pid` y `ttl_expires_at`.
    3. Si el PID está muerto (`kill -0 <pid>` falla) Y `ttl_expires_at` ya pasó:
       eliminar manualmente `rm ~/.kodo/locks/<repo-realpath>.lock` y documentar
       el incidente en lecciones aprendidas.
    4. Si el PID está vivo: NO eliminar — hay una sesión real corriendo.

    ### 5.3 Verdict del resolver dudoso antes de un launch

    `kodo gsd inspect <task-id>` — dry-run del resolver, no ejecuta side-effects.
    Salida con secciones `config`, `fetch`, `roadmap`, `match` (símbolos `✓`/`✗`).
    Exit codes (Phase 9 D-19): `0=ok`, `1=config error`, `2=fetch failure`.

    ### 5.4 Phase terminó y entró a Review

    `kodo gsd verify <session-id>` — el gate canónico. Ya cubierto en §4.2 con
    los exit codes deterministas (Phase 10 Pitfall #6 opción A). Nunca dupliques
    el comentario manual al provider.
