<decisions>
## Implementation Decisions

### Boundary skill.md ↔ prompt.md

- **D-01:** `skill.md` es la fuente canónica del comportamiento orquestador. Contiene el loop de supervisión completo, sección GSD (full y quick), flujos de diagnóstico CLI, mapping de proyectos, y mecanismo de auto-update.
- **D-02:** `src/orchestrator/prompt.md` se simplifica a un render mínimo provider-specific (mantiene los placeholders `{{provider_name}}` / `{{mcp_tool}}` / `{{provider}}` que `resolvePromptTemplate` ya sustituye al spawn). Debe permanecer **auto-suficiente como fallback degradado** cuando `kodo orchestrator` se lanza con `cwd ≠ repo` y por tanto la skill no se auto-carga.
- **D-03:** Ambos archivos cross-referencian al otro en su cabecera para que un lector que entra por cualquiera de los dos sepa dónde vive cada cosa.

### Disposición de la skill global

- **D-04:** `~/.claude/skills/kodo-orchestrate/skill.md` se elimina al cerrar la phase. La skill solo vive en `<repo>/.claude/skills/kodo-orchestrate/skill.md`.
- **D-05:** El contrato operativo queda: `kodo orchestrator` se invoca desde el repo kodo (`cwd = ~/dev/klab/kodo`) para que Claude Code auto-cargue la skill. Si se lanza desde otro cwd, la sesión arranca solo con `prompt.md` (caso degradado documentado, no error).
- **D-06:** Capturar D-05 explícitamente en `PROJECT.md > Constraints` al cerrar la phase (regla operativa nueva que afecta el deploy del orquestador).

### Provider-agnostic — estilo de redacción

- **D-07:** La skill se escribe en **lenguaje genérico narrativo** ("tu proveedor de tareas", "el MCP del provider configurado"). **Sin placeholders `{{...}}` literales** (Claude Code carga la skill cruda; no hay render).
- **D-08:** Primer paso del agente al arrancar: `cat ~/.kodo/config.json` para leer `provider` y derivar el namespace MCP por convención `mcp__<provider>__*`. Esto reemplaza el bloque actual hardcoded a Plane.
- **D-09:** Cero ejemplos Plane-specific en el cuerpo de la skill (no listar `mcp__plane__list_projects` etc.). Forward-compat para adapters GH/CU/local de v0.6+.

### CLI cheat-sheet y flujos diagnóstico

- **D-10:** Sección "Diagnóstico" task-oriented (síntoma → comando), con 4 flujos mínimos:
  - **Sesión stuck (>30min idle):** `cmux read-screen` para confirmar, luego `kodo logs --follow --session-of <task-id>` para ver actividad reciente; nudge vía `cmux send` o escalada según contexto.
  - **Lock no se libera tras Stop:** inspeccionar `~/.kodo/locks/<repo-realpath>.lock` (PID + TTL); si el PID está muerto y TTL expirado, eliminar manualmente y documentar.
  - **Verdict del resolver dudoso antes de un launch:** `kodo gsd inspect <task-id>` (dry-run del resolver, exit codes deterministas 0=ok, 1=config error, 2=fetch failure).
  - **Phase terminó y entró a Review:** `kodo gsd verify <session-id>` (gate canónico; el CLI hace `addComment + updateTaskState`; exit codes 0/1/2 + 64..78 para usage/not-found).
- **D-11:** No duplicar gate manual en comentarios al provider — el CLI es la única fuente para `gsd verify`. La sección reitera este invariante para sesiones full; explicita que las sesiones `[GSD quick]` se revisan manualmente (CLI no soporta verify en quick).

### Lecciones aprendidas

- **D-12:** Borrar las 4 entradas existentes de 2026-04-07 (`list_work_items 403`, `kodo:* UUIDs`, `liken ≡ s1s4_v2`, narrativa de primera sesión). Razones: tres son Plane-specific y contradicen la decisión provider-agnostic; la cuarta es anecdotal.
- **D-13:** Conservar el **mecanismo** de auto-update: la skill arranca con `## Lecciones aprendidas` vacío y mantiene las mismas instrucciones para que el agente añada entradas al cerrar (formato `- [fecha] descripción`). Si re-aparecen los hallazgos, el mecanismo los recaptura.

### Hook Stop / auto-commit del skill

- **D-14:** Fix de 2 líneas en `src/hooks/stop.js`:
  - L21: `SKILL_PATH = join(KODO_ROOT, '.claude', 'skills', 'kodo-orchestrate', 'skill.md')`.
  - L243: `git status --porcelain .claude/skills/`.
  - L255: `git add .claude/skills/ && git commit -m "skill: orchestrator learnings ${date}"`.
- **D-15:** El contrato auto-commit que la skill ya prometía vuelve a estar honrado. Se mantiene la línea en la skill que lo documenta ("el commit es automático — el hook Stop detecta cambios en `.claude/skills/`").
- **D-16:** Añadir tests del bloque `handleSkillAutoCommit` (hoy sin coverage porque el path muerto nunca disparaba). Mínimo: 2 escenarios — (a) cambios en `.claude/skills/` → commit; (b) sin cambios → no-op silencioso. Coherente con el patrón source-hygiene + spawnSync usado en Phases 16–17 (LOG-13..15, UAT-01..03).

### Mapping de proyectos

- **D-17:** Eliminar la sección "IDs conocidos" hardcoded de la skill actual (LIKEN/ROMAN/TENDERIO). La instrucción canónica es `cat ~/.kodo/projects.json` antes de cualquier launch, y "si no existe el mapping, preguntar al usuario antes de lanzar" (regla operativa ya respaldada por memoria — kodo_one_project_per_repo).

### Claude's Discretion

- Estructura concreta de secciones dentro de la skill (orden, headers) — pattern: arrancar por "Proceso de inicio" (cargado de estado + descubrimiento de tareas), luego "Reglas de operación", "Mapeo de proyectos", "Sesiones GSD" (full + quick), "Diagnóstico", "Cómo actualizar este skill", "Lecciones aprendidas". Coherente con la skill actual; el contenido es lo que cambia.
- Wording exacto del cross-reference entre skill.md y prompt.md (D-03).
- Si la sección "Reglas de operación" mantiene el detalle de selección de modelo (`Opus default, Sonnet si kodo:sonnet, Haiku si kodo:haiku`) o solo lo referencia a `config.claude` — preferible mantener el detalle textual porque es la regla más práctica del orquestador.
