## Deviations from Plan

### Bug en el acceptance criteria del plan (no-blocker)

**1. [Rule 1 - Bug en plan, no en código] Comando `node --input-type=module` del acceptance criteria usa `require` (CJS) en input ESM**
- **Found during:** Verificación de acceptance criteria al final de Task 1.
- **Issue:** El último acceptance criterion del plan ejecuta:
  ```
  node --input-type=module -e "import('./src/orchestrator/launch.js').then(m => { const t = require('fs').readFileSync(...); ... })"
  ```
  Con `--input-type=module`, `require` no está definido (Node 25.9.0 ESM puro) → `ReferenceError: require is not defined`. El test falla por la sintaxis del comando, no por una falla del prompt.md.
- **Fix:** Verifiqué el invariante semántico real (que `resolvePromptTemplate` no deje placeholders residuales) con la sintaxis ESM correcta:
  ```
  node --input-type=module -e "import { readFileSync } from 'node:fs'; import('./src/orchestrator/launch.js').then(m => { const t = readFileSync('./src/orchestrator/prompt.md','utf-8'); const out = m.resolvePromptTemplate(t, {provider:'plane'}); if(out.match(/\{\{[^}]+\}\}/)) process.exit(1); });"
  ```
  Resultado: `render OK — no placeholders left`. El opener renderiza como `Eres el orquestador de kodo. ... tareas de Plane.` confirmando que `{{provider_name}}` se sustituye correctamente.
