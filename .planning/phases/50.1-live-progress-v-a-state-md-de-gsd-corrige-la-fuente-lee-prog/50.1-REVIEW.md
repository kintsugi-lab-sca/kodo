## Summary

Phase 50.1 repoints the live-progress source from the (empty in real GSD sessions) `~/.kodo/progress/` surface to the `progress:` block of the GSD worktree's `STATE.md`, via a new `readGsdProgress`/`parseProgressBlock` leaf and the additive `computeRealWorktreePath` helper. The hook demotion of `task-progress.js` (50-02) is **clean**: the source and both of its test files were deleted, and no live `import`/`require` of `task-progress.js` remains anywhere in `src/` or `test/` (only historical comment mentions persist). SessionStart/Stop install/uninstall coverage is intact (`test/hooks/install.test.js` passes 5 behaviors). All 26 tests across the three reviewed test files pass.

The hard invariants hold: `src/server.js` is untouched (zero new endpoints), zero new deps (parser is hand-rolled with `node:fs`/`node:path` builtins), color isolation is preserved (no `picocolors` import in the dashboard), and `computeWorktreePath` (the 5-consumer legacy `.bg-shell` path) was not modified — `computeRealWorktreePath` is genuinely additive.

The mini-parser is **ReDoS-safe** as claimed: every regex is a compile-time constant, and the per-key `^\s+KEY:\s*(\d+)` pattern executes in sub-millisecond time even against a 200k-character adversarial input — there is no catastrophic backtracking and no regex is ever compiled from external input. The anti-traversal guard on `session_id` (`/`, `\`, `..` via `String.includes`) is present before path construction. `readGsdProgress` honours the never-throws contract (EACCES / corrupt / non-string content all degrade to `{status:'error'}`; ENOENT → `{status:'no-progress'}`).

The defects below are correctness/robustness gaps, not security holes. The most material is the parser **not scoping to the `progress:` block** (WR-01): any allowlisted key indented anywhere in the frontmatter is matched, so a stray `total_phases`/`completed_phases` under a different YAML block silently feeds the progress cell.

## Warnings

### WR-01: `parseProgressBlock` does not scope to the `progress:` block — matches allowlisted keys anywhere in the frontmatter

**File:** `src/cli/dashboard/progress.js:55-62`
**Issue:** The function's contract (docstring lines 36-40, 49-64) and the surrounding comments claim it extracts keys "del bloque progress:" indented "bajo `progress:`". It does not. The per-key regex `new RegExp(`^\\s+${key}:\\s*(\\d+)`, 'm')` matches the key indented under *any* parent in the frontmatter, with no anchoring to a `progress:` header. Verified empirically — a STATE.md whose frontmatter is:

```
---
other:
  total_phases: 42
  completed_phases: 7
---
```

(keys under an unrelated block, no `progress:` at all) yields `{status:'ok', n:7, m:42}` instead of `no-progress`. Any future GSD STATE.md that introduces a sibling block reusing one of these key names (`total_phases`, `completed_phases`, `total_plans`, `completed_plans`) will silently corrupt the dashboard's N/M. This is a latent correctness bug; the only thing masking it today is that GSD happens to use these names solely under `progress:`.

**Fix:** Isolate the `progress:` block before scanning keys, so only its indented children are considered. Keep it constant-regex / anti-ReDoS:

```js
export function parseProgressBlock(md) {
  const fm = FRONTMATTER_RE.exec(md);
  if (!fm) return null;
  // Slice the `progress:` block: from its header to the next top-level (non-indented) key or EOF.
  const blockRe = /^progress:\s*\r?\n((?:[ \t]+.*(?:\r?\n|$))*)/m; // constant, linear
  const blockHit = blockRe.exec(fm[1]);
  const body = blockHit ? blockHit[1] : ''; // no progress: block → no keys
  /** @type {Record<string, number>} */
  const out = {};
  for (const key of PROGRESS_KEYS) {
    const re = new RegExp(`^\\s+${key}:\\s*(\\d+)`, 'm');
    const hit = re.exec(body);
    if (hit) out[key] = Number(hit[1]);
  }
  return out;
}
```

Add a test asserting that keys under a non-`progress:` block are ignored.

### WR-02: `completed_phases > total_phases` is not clamped — renders a nonsensical `N/M` with N>M

**File:** `src/cli/dashboard/progress.js:96-98`
**Issue:** `n` and `m` are taken verbatim from the file. A STATE.md with `total_phases: 3` and `completed_phases: 5` returns `{status:'ok', n:5, m:3, completed:false}` (verified). `progCell` (format.js:245) renders this directly as `5/3`, which is both visually wrong and semantically impossible. It also defeats the `completed` flag: a 5/3 over-count never satisfies `n === m`, so a phase set that is actually finished-and-then-some would display as in-progress without the `✓`. STATE.md is GSD-maintained so this should not normally happen, but the leaf advertises never-throws/robust parsing of an external artifact and silently propagating an impossible ratio undermines that.

**Fix:** Clamp `n` to `[0, m]` (and treat `n >= m` as completed when `m > 0`):

```js
const m = block.total_phases;
const rawN = block.completed_phases ?? 0;
const n = Math.max(0, Math.min(rawN, m)); // clamp into [0, m]
return { status: 'ok', n, m, completed: m > 0 && n === m };
```

### WR-03: `deriveAnyProgress` is structurally always-true because the enrich assigns a non-null `progress` to every row (pre-existing, carried forward by the rewritten enrich)

**File:** `src/cli/dashboard/App.js:351-386` (consumed at `select.js:236-237`)
**Issue:** `deriveAnyProgress(rows)` returns `rows.some(r => r.progress != null)` and is intended (docstring select.js:221-235, App.js:384-386) to be a *structural* flag that hides the `prog` column when NO active session reports progress. But the enrich at App.js:351-379 assigns a non-null `progress` object to **every** row on every branch — non-GSD rows get `{ status: 'no-progress' }` (line 353), unusable rows likewise (line 366). Therefore `enriched` never contains a row with `progress == null`, so `anyProgress` is **always true** and the `prog` column never hides — verified by replaying the enrich shape against the predicate. This contradicts the stated D-06 contract.

Note: this is **pre-existing from Phase 50** (the prior enrich at 96b8f7b^ had the identical always-non-null shape and the same `deriveAnyProgress(enriched)` call). 50.1 did not introduce it, but the enrich block was rewritten in this phase and carries the defect forward, so it is in-scope for this review. Flagging as WARNING (cosmetic column-visibility, not a crash or data issue).

**Fix:** Either (a) gate the predicate on a meaningful progress status, or (b) keep `progress` null for rows with no real progress. Option (a) is the smaller diff:

```js
// select.js
export function deriveAnyProgress(rows) {
  return rows.some((r) => {
    const s = r.progress?.status;
    return s === 'ok' || s === 'error'; // a real reading, not the '—' placeholder
  });
}
```

If `deriveAnyGsd` should already gate the same column elsewhere, confirm the two flags are not redundant.

## Info

### IN-01: `total_plans` / `completed_plans` are parsed into the allowlist but never consumed

**File:** `src/cli/dashboard/progress.js:34` (declared), `progress.js:96-98` (only `total_phases`/`completed_phases` read)
**Issue:** `PROGRESS_KEYS` includes `total_plans` and `completed_plans`, so `parseProgressBlock` extracts them, but `readGsdProgress` only ever reads `total_phases`/`completed_phases`. The two plan keys are dead work — extracted then discarded. No correctness impact, but it implies a capability the code does not deliver and invites a future reader to assume plan-level progress is wired.
**Fix:** Either drop the two unused keys from `PROGRESS_KEYS` (smaller surface, faster), or add a comment at line 34 stating they are reserved/intentionally parsed for a future plan-granularity cell. Prefer dropping unless a concrete consumer is imminent.

### IN-02: Frontmatter must start at byte 0 — any leading whitespace/BOM degrades the whole file to `{status:'error'}`

**File:** `src/cli/dashboard/progress.js:30` (`FRONTMATTER_RE = /^---\r?\n.../`)
**Issue:** The frontmatter regex is anchored with `^` (without the `m` flag) so the `---` opener must be the very first bytes. A STATE.md with a leading blank line or a UTF-8 BOM yields `{status:'error'}` (verified with a leading `\n`), which `progCell` renders as `?` rather than the real N/M. GSD's generator currently writes `---` at byte 0, so this is latent, but it makes the "corrupt → error" classification fire on a benign formatting variation.
**Fix:** Tolerate optional leading whitespace/BOM by prepending an optional BOM escape (`\uFEFF?`) and `\s*` before the `---` opener — i.e. `/^\uFEFF?\s*---\r?\n([\s\S]*?)\r?\n---/`. Use the `` escape rather than a literal BOM so the source stays free of invisible characters. Keep it a compile-time constant (still anti-ReDoS).

### IN-03: Keep-last-good ref is keyed by `session_id`, but its `@type` JSDoc and the surrounding comments still describe the old `task_id` keying

**File:** `src/cli/dashboard/App.js:283-287` (typedef/comment) vs `App.js:370-376` (usage)
**Issue:** The ref's declaring comment (lines 283-285) says "Mapa por task_id → último {n,m,completed}", and the type annotation is `Map<string, {...}>`, but the enrich now `.set(sessionId, ...)` / `.get(sessionId)` (lines 370, 374). The keying changed from `task_id` (Phase 50) to `session_id` (50.1) but the comment was not updated. Functionally correct (both are unique per session), but the stale comment will mislead.
