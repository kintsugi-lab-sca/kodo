# Phase 50.1: Live-progress vía STATE.md de GSD (corrección de fuente) - Research

**Researched:** 2026-06-14
**Domain:** kodo dashboard (ink/React TUI) · lectura filesystem-style de artefactos GSD · hooks de Claude Code
**Confidence:** HIGH (todo el código relevante leído en esta sesión; el schema de `progress:` confirmado contra el generador real de GSD y contra el STATE.md propio de kodo)

## Summary

Phase 50 entregó un display correcto (columna condicional `prog`, 4 estados, cero-color, enrich client-side) pero alimentado por la fuente equivocada: el hook 50-02 recuenta `~/.claude/tasks/<sid>/`, que en sesiones GSD/kodo reales queda **vacío** porque GSD usa la herramienta `Agent` (subagentes), nunca `TaskCreate`. Phase 50.1 es quirúrgica: **cambia únicamente la fuente** que alimenta `readProgress`, leyendo el bloque `progress:` del frontmatter del `STATE.md` del worktree GSD, reutilizando todo el display intacto, y **elimina** el hook 50-02.

Tres hallazgos load-bearing confirmados con evidencia cruda esta sesión:
1. **El schema `progress:` es fiable y es disk-scan ground-truth.** El generador GSD (`bin/lib/state.cjs:782-879`) recomputa `completed_phases`/`total_phases` escaneando `.planning/phases/` en CADA `sync` (no acumula contadores frágiles). El STATE.md propio de kodo lo confirma: `total_phases:5, completed_phases:3`. Las keys son **condicionales** (`if (x !== null) progress.x = ...`) → un lector robusto debe tolerar keys ausentes.
2. **El bug `worktree_path` es real y confirmado en disco.** `computeWorktreePath` (`src/session/state.js:153`) devuelve `<projectPath>/.bg-shell/<sid>`, pero `claude --worktree <name>` crea un **git worktree** (verificado: `git worktree list` + `claude --help`), que se materializa en `<projectPath>/.claude/worktrees/<sid>`. El `.bg-shell/` real solo contiene `manifest.json`/`wave*-env.sh`, NO worktrees de sesión.
3. **CERO server.js.** `src/server.js:444-452` hace `...s` sobre cada SessionRecord → `gsd`, `worktree_path`, `project_path` ya llegan al cliente. El único punto de lectura del progreso es el enrich client-side en `App.js:347-366`. El invariante D-06/D-08 se preserva gratis.

**Primary recommendation:** Crear un lector nuevo `readGsdProgress(row)` (mold `readLightPlan`: síncrono, never-throws, anti-traversal) que (a) derive la ruta REAL del worktree con un helper nuevo `computeRealWorktreePath` SIN tocar `computeWorktreePath` (blast radius mínimo — DG-04), (b) lea `<worktree>/.planning/STATE.md`, (c) parsee el bloque `progress:` con un mini-parser hand-rolled (kodo NO tiene dependencia YAML), y devuelva `{status, n, m, completed}` con la misma forma que `readProgress` consume hoy. Repuntar `App.js:355` a este lector, gateado por `row.gsd === true`. Eliminar `task-progress.js` + su registro + sus 2 tests.

### Pattern 1: Lector filesystem-style never-throws (mold readLightPlan)
**What:** Función pura, síncrona, con DI para tests (HOME/path isolation), que mapea ENOENT/parse-fail a estados discriminados sin lanzar jamás.
**When to use:** Todo lector de artefacto del dashboard (es el contrato del leaf).
**Example:**
```javascript
// Source: src/cli/dashboard/progress.js:44-58 (readProgress) + plan.js:65-78 (readLightPlan)
// Forma a replicar para readGsdProgress:
export function readGsdProgress(worktreeBase, deps = {}) {
  const readFileFn = deps.readFileFn || ((p) => readFileSync(p, 'utf-8'));
  // worktreeBase ya validado anti-traversal por el caller (App.js, String.includes NO regex)
  try {
    const raw = readFileFn(join(worktreeBase, '.planning', 'STATE.md'));
    const prog = parseProgressBlock(raw);            // mini-parser, ver Pattern 2
    if (!prog || prog.total_phases == null) return { status: 'no-progress' };
    const n = prog.completed_phases ?? 0;
    const m = prog.total_phases;
    return { status: 'ok', n, m, completed: m > 0 && n === m };
  } catch (err) {
    const code = err?.code;
    if (code === 'ENOENT') return { status: 'no-progress' };  // sin STATE.md → '—'
    return { status: 'error' };                               // EACCES/corrupto → '?' + keep-last-good
  }
}
```

### Pattern 2: Mini-parser del bloque `progress:` (sin dependencia YAML)
**What:** El frontmatter GSD usa un bloque `progress:` indentado a 2 espacios con escalares enteros. NO se necesita un parser YAML completo — basta extraer el bloque y leer las keys numéricas conocidas.
**Example:**
```javascript
// El bloque real (verificado en /Users/alex/dev/klab/kodo/.planning/STATE.md):
//   progress:
//     total_phases: 5
//     completed_phases: 3
//     total_plans: 7
//     completed_plans: 7
//     percent: 60
//
// Mini-parser (anti-ReDoS: regex CONSTANTE sobre un bloque acotado, no derivada de input):
function parseProgressBlock(md) {
  // 1. Aislar el primer frontmatter (--- ... ---) — mismo enfoque que GSD stripFrontmatter.
  const fmMatch = md.match(/^---\r?\n([\s\S]*?)\r?\n---/);
  if (!fmMatch) return null;
  const fm = fmMatch[1];
  // 2. Leer keys conocidas con regex constante (NO compilada desde input).
  const out = {};
  for (const key of ['total_phases', 'completed_phases', 'total_plans', 'completed_plans']) {
    const m = fm.match(new RegExp(`^\\s+${key}:\\s*(\\d+)`, 'm')); // key es literal de allowlist → seguro
    if (m) out[key] = parseInt(m[1], 10);
  }
  return out;
}
```
**Nota anti-ReDoS:** las `key` son de una allowlist literal fija, no input externo — no hay vector ReDoS (espejo de la disciplina `/^\d+$/` constante en `plan.js:131`).

### Anti-Patterns to Avoid
- **Tocar `computeWorktreePath`:** rompe doctor (`gsd/doctor.js:151,164` escanea `.bg-shell` y cruza con `computeWorktreePath`) y dismiss. La función está acoplada a la lógica `.bg-shell` de housekeeping. Ver Runtime State Inventory.
- **Añadir un campo al payload de /status o un endpoint:** viola DG-06 (`git diff --quiet src/server.js` debe dar 0).
- **Parsear el STATE.md con `js-yaml`:** añade dependencia injustificada y rompe leaf-isolation.
- **Reusar el `percent` del bloque:** DG-01 fija N/M = fases, NO percent. El `percent` GSD aplica `min(plan_fraction, phase_fraction)` (`computeProgressPercent`) — semántica distinta, no usar.

## Common Pitfalls

### Pitfall 1: Confiar en `row.worktree_path` persistido
**What goes wrong:** El campo guardado apunta a `.bg-shell/<sid>` (inexistente como worktree) → el STATE.md nunca se encuentra → siempre `—`.
**Why it happens:** `computeWorktreePath` asumió mal la convención de `claude --worktree`.
**How to avoid:** Derivar la ruta con `computeRealWorktreePath(row.project_path, row.session_id)` en el punto de lectura. NUNCA usar `row.worktree_path` para localizar el STATE.md.
**Warning signs:** La columna `prog` muestra `—` para sesiones GSD que claramente tienen progreso.

### Pitfall 2: Asumir que todas las keys de `progress:` están presentes
**What goes wrong:** Crash o `NaN/NaN` si `completed_phases` falta.
**Why it happens:** El generador GSD añade cada key CONDICIONALMENTE (`state.cjs:874-878`: `if (x !== null) progress.x = ...`). Sesiones GSD muy tempranas (antes del primer disk-scan) o STATE.md legacy pueden tener el bloque parcial o ausente.
**How to avoid:** Tratar `total_phases` ausente como `no-progress` (→ `—`). `completed_phases` ausente → tratar como 0 (monótono desde 0). El mini-parser devuelve solo las keys que encuentra.
**Warning signs:** Evidencia: TENDERIO-9 tenía `total_phases:5, completed_phases:0` (válido, N=0). El STATE.md de kodo tiene los 4 + percent.

### Pitfall 3: Ancho de columna `prog` (DG-discretion)
**What goes wrong:** Header `prog` (4 chars) vs valores ahora pequeños (`3/5✓` = 4 chars, fases 0-9 típicas). `COLS.prog = 7` (`SessionTable.js:58`) se dimensionó para "N/M✓ peor caso de dobles dígitos" del modelo de TAREAS de Phase 50.
**Why it happens:** Phase 50 contaba tareas (podían ser decenas → `12/34✓` = 6 chars). Phase 50.1 cuenta FASES (típicamente 0-9 → `3/5✓` = 4 chars; peor caso realista `9/12✓` = 5 chars).
**How to avoid:** El planner decide si reducir `COLS.prog` de 7 a 5-6. CRÍTICO: el ancho está en una tabla golden-bytes-sensible (`test/dashboard-table.test.js`); cambiarlo requiere actualizar ese test. RECOMENDACIÓN: dejar `COLS.prog = 7` (cabe de sobra, cero churn de tests golden, el header `prog` ya ocupa 4) salvo que el planner quiera apretar el layout. Menor blast radius = no tocarlo.
**Warning signs:** Tests golden de tabla en rojo por mismatch de ancho.

### Pitfall 4: Romper el invariante CERO server.js
**What goes wrong:** Tentación de enriquecer el progreso server-side (donde ya se hace el `...s`).
**Why it happens:** El server ya itera sesiones; parece "natural" añadir ahí la lectura.
**How to avoid:** El enrich es CLIENT-SIDE en `App.js:347-366` (D-08 preservado). `git diff --quiet src/server.js` debe dar 0. El server solo pasa `...s` (que ya incluye `gsd`, `project_path`, `session_id`).
**Warning signs:** `git diff src/server.js` no vacío.

### Pitfall 5: Lectura síncrona en el render con coste real de disco
**What goes wrong:** El enrich corre en CADA render (`App.js:347`), leyendo N ficheros STATE.md síncronos. Con muchas sesiones GSD podría notarse.
**Why it happens:** El mold readProgress/readLightPlan es síncrono por diseño (sin server, sin await).
**How to avoid:** Es el patrón ya aceptado en Phase 44/50 (readPlan/readProgress síncronos en render). El gate `row.gsd === true` limita la lectura solo a filas GSD. STATE.md es un fichero pequeño. ACEPTABLE para v1 (mismo perfil que el legacy). Si molesta: memoizar por (session_id, mtime) — DEFERRED, no necesario.
**Warning signs:** Lag perceptible del dashboard con muchas sesiones GSD simultáneas.
