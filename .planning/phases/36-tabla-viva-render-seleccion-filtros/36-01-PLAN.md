## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server `/status` JSON → dashboard derive layer | Server-provided session fields (`summary`, `task_ref`, `project_name`, `phase_id`) cross into render-bound strings. Read-only; no mutation, no auth, no network egress from the TUI. |
| operator keyboard → filter query | In-memory filter string typed by the operator (the only user input in the phase); consumed by `parseFilter`/`applyFilter`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-36-01 | Tampering / DoS | `applyFilter` / `parseFilter` filter query | mitigate | Substring matching via `String.includes` ONLY; NEVER `new RegExp(query)`. Avoids ReDoS / regex injection from operator-typed filter text (RESEARCH Security V5). Enforced by the literal-special-char test in dashboard-filter.test.js + grep gate in acceptance criteria. |
| T-36-02 | Spoofing / Tampering | server-provided string fields rendered in rows | accept (this plan) | This plan produces display STRINGS only (no rendering). ink `<Text>` (Plan 02) renders children literally and does not interpret embedded escape sequences as control codes — documented assumption. Defense-in-depth CSI stripping is deferred to Plan 02's `rowCells` rendering decision; risk is LOW (task data originates from the operator's own provider). |
| T-36-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this phase — all deps (ink@6.8.0, react@19.2.6, ink-testing-library@4.0.0) already present and exercised by green Phase 34/35 suite (verified `npm ls`). RESEARCH Package Legitimacy Audit: zero new packages. |
