<decisions>
## Implementation Decisions

### TUI-17 — Blocker host↔TUI (Phase 38 dead code)
- **D-01:** **Eliminar** el código muerto del wiring host, NO cablearlo. Rationale: el camino del server (`reconcileTick` → `state.json` → `GET /status` → tabla) ya entrega los badges idle/needs-input al operador, cumpliendo la promesa funcional de Phase 38. Cablear la consulta host directa desde el dashboard introduciría una **segunda fuente de verdad** para `alive`/estados — exactamente la divergencia que TUI-13/14 corrige. Simplicidad + una sola fuente de verdad.
- **D-02:** Borrado concreto a aplicar (el planner verifica los sitios exactos):
  - `src/cli/dashboard/index.js`: quitar la construcción/paso de la prop `host` (`getHost('cmux', {binary})` + `host` en el objeto de props a `App`). Eliminar el import dinámico `../../host/interface.js` si queda sin uso.
  - `src/cli/dashboard/App.js`: eliminar el estado `hostError`/`setHostError`, su dismiss en el handler de input, el paso de `hostError` al render, y las constantes exportadas `HOST_ERR_UNAVAILABLE` / `HOST_ERR_TIMEOUT`.
  - `src/cli/dashboard/SessionTable.js`: quitar el import de las constantes `HOST_ERR_*` (importadas para "matar el drift").
- **D-03:** El blocker se cierra **documentando** en la verificación que el camino del server cumple la promesa funcional; no se introduce consulta host directa.

### TUI-13/14 — Fuente única de `alive`
- **D-04:** En `GET /status` (`src/server.js`, bloque de enrich ~línea 379-383) **dejar de sobreescribir** el campo `alive`. El valor autoritativo es el que `reconcileTick` ya escribió en `state.json` (vía WorkspaceHost `listWorkspaces() --json`). Se elimina el legacy `alive: workspaceList.includes(s.workspace_ref)`. El cómputo de `elapsed_min` se preserva.
- **D-05:** `displayStatus` (`src/server.js:199-200`, que deriva `dead`/`idle` a partir de `!s.alive`/`s.alive`) debe consumir el **mismo** `alive` autoritativo. Verificar que no quede leyendo un valor legacy/recomputado.
- **D-06:** Guard de Enter (`src/cli/dashboard/App.js:418`) — **fail-open**: se preserva el comportamiento actual (solo bloquea con `row.alive === false`). Si `alive` llega `undefined` (reconcileLoop aún no tickeó, o server sin host), Enter **permite** el attach; si el workspace ya no existe, el fallo real lo captura `FOCUS_ERR_ZOMBIE`/`FOCUS_ERR_ENOENT`. Coherente con la degradación elegante del milestone; no se castiga al operador por un tick aún no llegado.

### TUI-15 — Overlay de comentarios: unsupported vs vacío
- **D-07:** El server señala la diferencia con un campo **aditivo** en la respuesta de `/comments/{taskId}` (NO se crea endpoint nuevo — solo cambia la shape): cuando `typeof provider.listComments !== 'function'`, responder `{ comments: [], supported: false }`. Cuando el provider sí soporta y devuelve vacío: `{ comments: [], supported: true }` (o sin el campo). Byte-compatible: clientes viejos ignoran `supported`.
- **D-08:** La TUI renderiza mensajes distintos: `supported === false` → mensaje tipo "comments not supported by this provider"; `supported !== false` y lista vacía → el mensaje vacío actual (`OVERLAY_COMMENTS_EMPTY`, "no comments yet"). El planner localiza la constante real en el dashboard ink.
- **D-09:** Verificar si la vista HTML del server (`src/server.js:168-182`, render `box.innerHTML` con su propio "no comments") necesita el mismo trato, además del overlay ink del `kodo dashboard`. Si es una superficie viva, aplicar la misma distinción; si es legacy/no consumida, dejar nota.

### Alcance / bookkeeping
- **D-10:** 39.1 = **4 gaps de código + bookkeeping ligero y verificable**. El bookkeeping ligero incluye: reconciliar el checkbox de Phase 34 en el roadmap/REQUIREMENTS, marcar `[x]` TUI-15/TUI-16, y poner la columna Status de la traceability table en estado coherente con las VERIFICATION passed.
- **D-11:** La auditoría retroactiva pesada (VERIFICATION.md de 37/38, VALIDATION.md Nyquist de 38/39, frontmatter SUMMARYs) se **delega** a `/gsd-complete-milestone` / `/gsd-validate-phase`, que son sus herramientas dedicadas. No se mezcla con el trabajo de código de esta fase.

### Claude's Discretion
- **TUI-10 (statusColor v3):** Fix mecánico, sin gray area. `statusColor(status, alive)` (`src/cli/dashboard/format.js:91`) solo ve `session.status`; debe derivar también del estado v3 (`session.state`) para colorear la columna `status` de sesiones idle/needs-input cuyo `status` v2 es null. Patrón ya establecido por `stateBadge` (`format.js:136`, que lee `session.state ?? session.status`). El planner elige la firma exacta (p. ej. pasar `state` además de `status`/`alive`) preservando la paleta LOCKED D-08 (nombres de color ink, no ANSI) y el byte-determinismo de `--json`/NO_COLOR.
- Estructura de tests: seguir el patrón de aislamiento existente (format-isolation, dispatcher-isolation) para cualquier invariante source-hygiene que toquen estos cambios.
