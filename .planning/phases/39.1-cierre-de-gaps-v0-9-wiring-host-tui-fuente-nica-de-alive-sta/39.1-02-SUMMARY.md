# Phase 39.1 Plan 02: Fuente única de verdad para `alive` en GET /status Summary

`GET /status` deja de recomputar `alive` con el check legacy `cmux.listWorkspaces()` y lo pasa-through desde el valor autoritativo que `reconcileTick` escribe en `state.json`, cerrando la divergencia que podía colar focus a un workspace ya marcado muerto por el host.

## What Was Built

- **Task 1 (D-04):** En el handler `GET /status` (`src/server.js`), se eliminó el bloque de enrich legacy que recomputaba `alive`:
  - Borradas las líneas `let workspaceList = '';` y `try { workspaceList = await cmux.listWorkspaces(); } catch {}`.
  - Eliminada la propiedad `alive: workspaceList.includes(s.workspace_ref)` del objeto del `.map`.
  - El objeto enriquecido queda como `{ ...s, elapsed_min: ... }` — `...s` ya trae el `alive` autoritativo de `state.json` (escrito por `reconcileTick`, único escritor vía `startReconcileLoop`).
  - `elapsed_min` preservado (se sigue computando en el server).
  - Comentario actualizado para reflejar que ahora solo se computa `elapsed_min` y que `alive` viene de `state.json`.
  - `cmux.rename`/`cmux.setColor` (server.js:485-486) intactos — el import/handle de `cmux` se preserva, solo se eliminó el consumidor `listWorkspaces`.
  - Commit: `ecab37c`

- **Task 2 (D-05/D-09 — verificación, sin edición):** Verificado que `displayStatus` (`src/server.js:198-200`, JS de NAVEGADOR embebido como template string — la vista HTML del dashboard web, NO código Node server-side) solo **deriva** `dead`/`idle` de `s.alive`/`s.status`/`s.elapsed_min`, sin recomputar `alive`:
  - `displayStatus = 'dead'` cuando `!s.alive && s.status === 'running'`
  - `displayStatus = 'idle'` cuando `s.alive && s.status === 'running' && s.elapsed_min > 30`
  - Tras la Task 1, el `s.alive` que ve `displayStatus` ES el autoritativo → **D-05 satisfecho automáticamente por D-04**. No se editó el template string (no lee ningún campo legacy distinto de `s.alive`).

## Decisions Made

- **D-04 — GET /status deja de sobreescribir `alive`:** El valor autoritativo es el de `reconcileTick` (`reconcile.js:203` / `state.js:123`), escrito en `state.json` por el único escritor (`startReconcileLoop`). Dejar de pisarlo en `/status` elimina el segundo cómputo divergente. Cambio sustractivo byte-compatible: el campo `alive` sigue presente en la respuesta, solo cambia su origen al autoritativo.
- **D-05 — displayStatus consume el `alive` autoritativo:** Satisfecho por construcción tras D-04. `displayStatus` solo deriva la etiqueta visual `dead`/`idle`, nunca recalcula `alive`.

None — el plan se ejecutó exactamente como fue escrito. La documentación de Task 2 vive en este SUMMARY, como pide el plan.

## Notas / NOTA D-09

La vista HTML web (`renderSession`/`displayStatus`, JS de navegador embebido en `src/server.js`) **reimplementa la heurística `idle` en el cliente** con `s.elapsed_min > 30`, una clasificación divergente del estado v3 que el server ya conoce (el server no expone un `displayStatus` derivado; cada superficie lo recalcula). Esto es la **vista HTML legacy (web)**, NO el overlay ink del `kodo dashboard`. Siguiendo el mismo criterio defensivo que D-09, se deja como NOTA sin alinearla en esta fase: queda fuera del alcance prioritario (el overlay ink es la superficie viva de v0.9) salvo que se confirme que esta vista HTML es superficie viva crítica. El riesgo es cosmético/de presentación, no afecta al `alive` autoritativo (que ya es fuente única tras D-04) ni al guard de Enter del dashboard ink.
