## Directory Purposes

**`bin/`:**
- Purpose: CLI entry point
- Contains: Single executable script `kodo`
- Key files: `kodo` (shebang Node.js script that imports `src/cli.js`)

**`src/`:**
- Purpose: All application code
- Contains: 15 files implementing core logic, integrations, hooks, state management
- Key files: `cli.js` (command router), `server.js` (webhook), `config.js` (settings)

**`src/plane/`:**
- Purpose: Plane CE API integration
- Contains: REST client with work item, project, state, label, comment operations
- Key files: `client.js` (main PlaneClient class)

**`src/cmux/`:**
- Purpose: cmux terminal multiplexer wrapper
- Contains: CLI invocation helpers, color mapping
- Key files: `client.js` (execFile wrapper), `colors.js` (status enums)

**`src/session/`:**
- Purpose: Session lifecycle management
- Contains: State persistence, workspace launching, health monitoring
- Key files: `state.js` (JSON store), `manager.js` (launcher), `health.js` (monitors)

**`src/hooks/`:**
- Purpose: Claude Code hooks for context and completion handling
- Contains: Executable scripts that integrate with Claude Code
- Key files: `session-start.js` (context injection), `stop.js` (completion), `install.js` (registration)

**`src/orchestrator/`:**
- Purpose: Intelligent multi-session supervisor
- Contains: Orchestrator Claude session launcher and system prompt
- Key files: `launch.js` (launcher), `prompt.md` (system instructions)

**`test/`:**
- Purpose: Unit test suite
- Contains: Tests for state management and label parsing
- Key files: `state.test.js`, `labels.test.js`

**`skills/`:**
- Purpose: Knowledge base for orchestrator
- Contains: Auto-learned quirks, project mappings, validated processes
- Key files: `kodo-orchestrate/skill.md` (accumulated during orchestrator sessions)

## Key File Locations

**Entry Points:**
- `bin/kodo`: Executable CLI entry — runs `src/cli.js` via shebang
- `src/cli.js`: Command definitions and routing using Commander.js
- `src/server.js`: HTTP server on port 9090, webhook handler

**Configuration:**
- `src/config.js`: Loads `~/.kodo/config.json`, `~/.kodo/projects.json`, `~/.kodo/.env`
- `DEFAULT_CONFIG` object in `config.js`: All default settings with comments

**Core Logic:**
- `src/session/manager.js`: Work item → Claude launch pipeline
- `src/session/state.js`: Session CRUD operations, `~/.kodo/state.json` persistence
- `src/check.js`: Health check logic, orchestrator trigger conditions

**Integration Points:**
- `src/plane/client.js`: All Plane API calls (9 public methods)
- `src/cmux/client.js`: All cmux CLI calls (6 public functions)
- `src/labels.js`: Label parsing (2 public functions)

**Testing:**
- `test/state.test.js`: Session state CRUD tests
- `test/labels.test.js`: Label parsing unit tests
- Run tests: `npm test`

## Naming Conventions

**Files:**
- JavaScript modules: camelCase (`sessionManager.js`, `planeClient.js`)
- Hooks: Hyphenated descriptive names (`session-start.js`, `stop.js`)
- Test files: Suffix `.test.js` (not `.spec.js`)
- Markdown docs: UPPERCASE (`README.md`, `ARCHITECTURE.md`)

**Directories:**
- Feature modules: Descriptive plural or singular noun (`plane`, `cmux`, `session`, `hooks`, `orchestrator`)
- Flat structure preferred; no deeply nested hierarchy

**Functions/Exports:**
- Command handlers: camelCase `loadConfig()`, `startServer()`
- Async functions: Return `Promise<T>`
- Private helpers: prefixed in comments or JS `#private` fields (not used yet)
- Classes: PascalCase (`PlaneClient`)

**Variables:**
- Constants: UPPER_SNAKE_CASE (`KODO_DIR`, `STATE_PATH`, `DEFAULT_CONFIG`)
- Session objects: descriptive properties (`plane_id`, `workspace_ref`, `started_at`)
- Config keys: snake_case (`workspace_slug`, `trigger_state`, `max_parallel`)

## Where to Add New Code

**New CLI Command:**
1. Add command definition in `src/cli.js` (program.command('name'))
2. Import and implement handler from appropriate module (e.g., `src/session/manager.js`)
3. Document in README.md
4. Test via `bin/kodo name`

**New Feature (Session Integration):**
1. If querying Plane: Add method to `src/plane/client.js` (PlaneClient class)
2. If controlling cmux: Add function to `src/cmux/client.js` module
3. If affecting sessions: Update `src/session/manager.js` or `src/session/health.js`
4. If state changes: Update session record shape in `src/session/state.js` typedef
5. Update tests in `test/` correspondingly

**New Hook:**
1. Create file in `src/hooks/` (e.g., `src/hooks/new-event.js`)
2. Make it executable (`#!/usr/bin/env node` shebang)
3. Implement stdin → stdout pattern (read JSON, output JSON via `hookSpecificOutput`)
4. Register in `src/hooks/install.js` under appropriate hook list
5. Test by running `kodo install` and verifying in `~/.claude/settings.json`

**New Integration (External API):**
1. Create module in appropriate directory (e.g., `src/github/client.js`)
2. Implement client class following `PlaneClient` pattern (constructor with config, `request()` method, specific endpoints)
3. Add to `config.js` if new credentials needed
4. Import and use in command handlers or session logic

**Utilities:**
- Shared helpers: Add to `src/config.js` (env/config related) or create new module like `src/utils.js`
- Date/time helpers: Consider adding to `src/cli.js` (already has `timeSince()`)
- String formatters: Add inline or to new `src/format.js`

**Tests:**
- Unit tests: `test/moduleName.test.js`
- Use Node.js built-in `node --test` runner
- No test framework dependencies beyond assertions

## Special Directories

**`~/.kodo/` (User home config):**
- Purpose: User credentials and project mappings
- Generated by: `kodo config` command
- Committed: No (gitignore by user)
- Files:
  - `.env`: `PLANE_API_KEY`, `PLANE_WEBHOOK_SECRET` (secrets, not committed)
  - `config.json`: Plane URL, workspace slug, thresholds, colors
  - `projects.json`: Project ID → local filesystem path mappings
  - `state.json`: Active sessions tracking
  - `server.pid`: Running webhook server PID (temporary)

**`.planning/codebase/` (GSD analysis):**
- Purpose: Machine-readable architecture and structure docs
- Generated by: GSD `/gsd:map-codebase` tool
- Committed: Yes (checked into git for reference)
- Files: `ARCHITECTURE.md`, `STRUCTURE.md`, `CONVENTIONS.md`, `TESTING.md`, `STACK.md`, `INTEGRATIONS.md`, `CONCERNS.md`

**`skills/kodo-orchestrate/` (Orchestrator knowledge):**
- Purpose: Accumulated learning from orchestrator sessions
- Generated by: Orchestrator session → stop hook auto-commit
- Committed: Yes (auto-committed on each orchestrator session)
- Files: `skill.md` (markdown notes on quirks, mappings, decisions)

**`test/` (Test suite):**
- Purpose: Unit tests for core modules
- Generated: Developer-written
- Committed: Yes
- Run: `npm test`
