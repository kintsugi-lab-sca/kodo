## Summary

La fase migra la skill `kodo-orchestrate` al repo, arregla los paths del hook Stop
(`SKILL_PATH` y el glob de `git status`) y añade cobertura del auto-commit. El
trabajo está hecho con cuidado y la mayor parte de las invariantes declaradas en
el contexto se cumplen: el override `KODO_ROOT` es aditivo, el switch
`buildStopNudgeText` queda intacto, el patrón de aislamiento del test
(mkdtempSync + git config local) es el correcto, y la skill es genuinamente
provider-agnostic.

Dicho esto, hay **un BLOCKER documental** que rompe el contrato self-describing de
la propia skill (referencia a una función inexistente que el agente intentará
usar) y varios WARNINGS que deberían arreglarse antes de cerrar la fase: el
mensaje de commit no firma "Co-Authored-By" pero sí pasa por `git commit` que en
la máquina del dev tiene `gpg.commitsign=true` global — el código de producción
no desactiva firma localmente como sí hace el test, así que en producción se
puede colgar pidiendo passphrase GPG. También hay incoherencia en el
contrato `prompt.md ↔ skill.md` sobre el verdict `missing/malformed` y la
descripción de exit codes.

## Critical Issues

### CR-01: Skill referencia una función inexistente (`handleSkillAutoCommit`)

**File:** `.claude/skills/kodo-orchestrate/skill.md:172`
**Issue:** La skill documenta:
> "el hook Stop (`src/hooks/stop.js`) detecta cambios en `.claude/skills/` y los committea al terminar la sesión orquestadora vía `handleSkillAutoCommit`."

Pero la función real en `src/hooks/stop.js:238` se llama `handleOrchestratorStop`. No existe `handleSkillAutoCommit` en todo el repo (verificado con `grep -rn`). Esto rompe el contrato self-describing: un agente que lea la skill y vaya a buscar la función para entender el mecanismo no la encontrará y, peor, podría intentar invocarla por nombre. Es además una invariante explícita en el plan D-15 ("el contrato auto-commit que la skill ya prometía vuelve a estar honrado") — el contrato es exactamente lo que se rompe aquí, no en el code path sino en la documentación.

**Fix:**
```diff
- **El commit es automático** — el hook Stop (`src/hooks/stop.js`) detecta
- cambios en `.claude/skills/` y los committea al terminar la sesión
- orquestadora vía `handleSkillAutoCommit`. No necesitas hacer `git commit`
- manualmente; solo edita el archivo y deja que el hook haga el resto.
+ **El commit es automático** — el hook Stop (`src/hooks/stop.js`) detecta
+ cambios en `.claude/skills/` y los committea al terminar la sesión
+ orquestadora vía `handleOrchestratorStop`. No necesitas hacer `git commit`
+ manualmente; solo edita el archivo y deja que el hook haga el resto.
```

## Warnings

### WR-01: `handleOrchestratorStop` puede colgarse en producción por prompt GPG interactivo

**File:** `src/hooks/stop.js:255`
**Issue:** El test desactiva explícitamente `commit.gpgsign false` en el repo tmpdir
(test/skill-auto-commit.test.js:61) porque sabe que el dev tiene `gpg.commitsign=true`
global. En producción, `handleOrchestratorStop` invoca:
```js
execSync(`git add .claude/skills/ && git commit -m "..."`, { cwd: KODO_ROOT });
```
sin pasar `-c commit.gpgsign=false` ni `--no-gpg-sign`. Si GPG pide passphrase
(pinentry-curses sin TTY, o gpg-agent caducado), `execSync` se cuelga **dentro
del hook Stop de Claude Code**, bloqueando el cierre del orchestrator. El
`try/catch` exterior atrapa errores pero no timeouts de pinentry. El test
sortea esto silenciando GPG en el repo aislado, así que el contrato real nunca
se ejercita con la configuración del dev.

Side-note: incluso si pasa, el commit acaba **firmado** con la clave personal
del dev y atribuido a su `user.email` — eso es lo deseado, pero hay que ser
consciente de que el hook estampa firma personal sobre cambios que escribió
el LLM.

**Fix:** Pasar flag para no firmar y/o usar un timeout duro. Mínimo invasivo:
```js
execSync(
  `git -c commit.gpgsign=false add .claude/skills/ && git -c commit.gpgsign=false commit -m "skill: orchestrator learnings ${date}"`,
  { cwd: KODO_ROOT, encoding: 'utf-8', timeout: 10_000 },
);
```
O preferiblemente partirlo en dos execFileSync con args array (evita el `&&`
en shell y la composición de comandos) — ver WR-02.

### WR-02: `execSync` con string compuesto + `${date}` permite inyección si la fuente cambia

**File:** `src/hooks/stop.js:255`
**Issue:** Hoy `date` viene de `new Date().toISOString().slice(0, 10)` → seguro.
Pero el patrón (`execSync` con string interpolado + shell `&&`) es frágil ante
futuros refactors. Si mañana alguien decide incluir el `session.task_ref` o un
título en el mensaje de commit, se abre command injection trivial (cualquier
backtick o `$()` o `"`/`\` mal escapado ejecuta arbitrario en `KODO_ROOT`).
Adicionalmente, la concatenación con `&&` significa que un fallo de `git add`
(p.ej. permisos) deja el árbol potencialmente staged sin commit.

**Fix:** Usar `execFileSync` con args array y separar las dos operaciones:
```js
import { execFileSync } from 'node:child_process';
execFileSync('git', ['-c', 'commit.gpgsign=false', 'add', '.claude/skills/'], {
  cwd: KODO_ROOT, encoding: 'utf-8', timeout: 10_000,
});
execFileSync('git', [
  '-c', 'commit.gpgsign=false', 'commit',
  '-m', `skill: orchestrator learnings ${date}`,
], { cwd: KODO_ROOT, encoding: 'utf-8', timeout: 10_000 });
```

### WR-03: `prompt.md` describe el verdict `missing/malformed` que la skill canónica NO menciona

**File:** `src/orchestrator/prompt.md:34` y `.claude/skills/kodo-orchestrate/skill.md:86-91`
**Issue:** `prompt.md` enumera 4 verdicts: `pass`, `fail`, `missing`,
`malformed`, y exit codes `0/1/2 + 64..78`. La skill (canónica, según D-01)
enumera otros: `pass (0)`, `soft-fail (1)`, `hard-fail (2)`, `usage / not-found
(64..78)`. La taxonomía es **incoherente**:

- `prompt.md`: `missing` y `malformed` mapean a exit code 2.
- `skill.md`: exit code 2 es "hard-fail (rompe must-haves)".

Esto no es un detalle de wording — son contratos de verdicts distintos y el
orchestrator decide rondas en función de ellos. En el modo degradado (cwd ≠
repo) el agente usa `prompt.md`; en modo nominal usa `skill.md`. Mismo CLI, dos
interpretaciones del exit code 2. El contexto del usuario reconoce esto como
"hotfix tras tests pre-existentes PM3/PM4" — los tests dictan el wording del
prompt, pero la skill **debe alinearse**, no quedar desfasada.

**Fix:** Decidir cuál es la realidad del CLI (consultar `src/gsd/verify.js`) y
alinear ambos textos. Si el CLI realmente emite verdicts `missing` y
`malformed` además de `fail`, la skill debe listarlos. Si no, retirarlos del
prompt y re-evaluar los tests PM3/PM4.

### WR-04: Tests escriben en `process.cwd()` derivado, no validan que el archivo committeado sea el esperado

**File:** `test/skill-auto-commit.test.js:141-156`
**Issue:** El test A verifica:
- `result.status === 0`,
- subject del último commit con regex `^skill: orchestrator learnings \d{4}-\d{2}-\d{2}$`,
- working tree limpio,
- count of commits +1,
- HEAD avanzado.

No verifica que el archivo que se appendeó (`.claude/skills/kodo-orchestrate/skill.md`
con la línea `- [2026-05-11] new learning`) sea **realmente el que entró en el commit**.
Si un bug futuro hiciera que `git add .claude/skills/` se cambiase por `git add
-A` (incidente histórico documentado en `kodo_no_git_add_all.md`), el test
seguiría pasando, pero el commit incluiría toda la pizza del working tree.

**Fix:** Añadir aserción sobre el diff del commit:
```js
const diffFiles = git(tmpRepo, 'diff --name-only HEAD~1 HEAD').split('\n').sort();
assert.deepEqual(
  diffFiles,
  ['.claude/skills/kodo-orchestrate/skill.md'],
  'commit only includes the skill file (regression for git add -A)',
);
```

### WR-05: El test ignora el caso "orchestrator session con cambios fuera de `.claude/skills/`"

**File:** `test/skill-auto-commit.test.js` (todo el archivo)
**Issue:** Cubre (A) cambios en skills → commit y (B) sin cambios → no-op. No
cubre el caso (C) **cambios fuera de skills/** (p.ej. el dev tocó
`src/foo.js` mientras la orchestrator session estaba abierta) → el hook debe
**no committear** y dejar esos cambios donde están. Hoy el código lo hace
correctamente porque `git status --porcelain .claude/skills/` filtra por path,
pero sin test ese comportamiento es regression-prone.

Recordatorio del incidente histórico `kodo_no_git_add_all.md`: "sin worktree,
prohibido `git add -A` / `commit -a`; arrastra staging de otra sesión".

**Fix:** Añadir un tercer caso:
```js
it('C: leaves changes outside .claude/skills/ untouched', () => {
  ({ tmpHome, tmpRepo } = makeIsolatedRepo());
  writeFileSync(join(tmpRepo, 'unrelated.txt'), 'dev work in progress', 'utf-8');
  const result = runStopHookChild({ tmpHome, tmpRepo, sessionId: 'orch-C' });
  assert.equal(result.status, 0);
  const status = git(tmpRepo, 'status --porcelain');
  assert.match(status, /^\?\? unrelated\.txt$/, 'unrelated changes preserved');
  // No commit creado (no había cambios en skills/)
  assert.equal(git(tmpRepo, 'log --format=%s -n 1'), 'initial');
});
```

### IN-03: El switch de `buildStopNudgeText` deja escape literal `\\n` en plantillas template literal

**File:** `src/hooks/stop.js:46, 50, 54`
**Issue:** Los tres returns terminan en `\\n` dentro de template literal, lo

El comentario en L45 dice "Escape literal `\\n` preservado (D-04 Phase 10)" —

### IN-04: `tmpHome` y `tmpRepo` declarados con `let` outer-scope son frágiles ante test failure

**File:** `test/skill-auto-commit.test.js:107-115`
**Issue:** Si la primera línea del `it(...)` (`{ tmpHome, tmpRepo } = makeIsolatedRepo();`) **lanza** (p.ej. `mkdtempSync` falla por `EMFILE`), las variables quedan undefined, el `afterEach` salta el rmSync por la guarda, y el test reportará la excepción de `makeIsolatedRepo`. OK. Pero si la asignación **parcial** falla a mitad (improbable pero posible si se añadiera más init), `tmpHome` podría quedar set y `tmpRepo` undefined — orden de creación en `makeIsolatedRepo` es secuencial (`tmpHome` primero, L46), así que un fallo en `mkdtempSync` del repo dejaría un tmpHome huérfano sin cleanup.

Microscópico, pero el patrón canónico de Phase 16-17 que el archivo cita usa try/finally dentro de cada `it` precisamente por esto.

**Fix:** Mover la creación dentro de try/finally local al `it`, o crear ambos dirs y solo asignarlos al outer scope tras éxito completo:
```js
function makeIsolatedRepo() {
  const tmpHome = mkdtempSync(...);
  try {
    const tmpRepo = mkdtempSync(...);
    // ...resto del init...
    return { tmpHome, tmpRepo };
  } catch (e) {
    rmSync(tmpHome, { recursive: true, force: true });
    throw e;
  }
}
```
