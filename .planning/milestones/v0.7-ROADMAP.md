# Milestone v0.7: GitHub Issues Adapter

**Status:** ✅ SHIPPED 2026-05-14
**Phases:** 23-27 (5 phases)
**Total Plans:** 11 (12 including 23-03 optional/skipped)

## Overview

Milestone v0.7 valida la promesa arquitectónica de v0.2 (provider abstraction) implementando GitHub Issues como **segundo adapter** funcional, junto al adapter Plane existente. Entrega cuatro capacidades:

1. **GitHub adapter**: `GitHubClient` (REST + auth PAT + rate limit + etag/304) → `createGitHubProvider` (9 métodos canónicos `TaskProvider`) → `normalizeIssue` (shape simétrico con Plane).
2. **Polling trigger channel**: tercer canal junto a webhook + manual CLI; loop async con `clock` injectable, state cache atómico (`~/.kodo/polling-state.json` con etag preservado en 304), retry exponencial fail-open, fire-and-forget dispatch delegando idempotencia al lock per-repo Phase 8.
3. **Config wizard + CLI**: `kodo config` reconoce `provider: github` con auto-detect del git remote; `kodo polling start/stop/status` daemon con PID file atómico + exit codes deterministas; `kodo orchestrate --polling` integrado con mutex implícito documentado.
4. **Cross-provider contract matrix**: `test/providers/contract.test.js` itera Plane + GitHub × 7 asserts core = 14 nuevos casos demostrando empíricamente que cambiar de provider NO requiere reescribir lógica.

El invariante v0.2 queda **empíricamente validado** con código real (no solo aspiracional). Test suite cierra v0.6=688 → v0.7=777 (+89 nuevos, 0 fail, 0 new skips). Audit: `v0.7-MILESTONE-AUDIT.md` (status `passed`).

## Milestone Summary

### Key Decisions

1. **D-01 method-name fantasy corrección (Phase 24 CONTEXT.md):** El roadmap inicial listaba `listTasks/listLabels/listStates/transitionTask` para `GitHubProvider`. El contrato canonical en `src/interface.js` (`TASK_PROVIDER_METHODS`) son 9 nombres distintos. Corrección aplicada antes de implementación; doc safety net en Phase 24-03 previene reaparición.
2. **D-17 mutex implícito vía lock per-repo (Phase 26 CONTEXT.md):** Daemon path (`kodo polling start`) y orchestrator-integrated path (`kodo orchestrate --polling`) son ortogonales; el lock per-repo Phase 8 GSD-10 garantiza la propiedad esencial sin requerir mutex explícito. Simplicity-first (Karpathy Rule 2). Documentado en `--help` text.
3. **D-08 zero breaking change schema (Phase 26 CONTEXT.md):** `DEFAULT_CONFIG` NO modificado para añadir `providers.github`. El wizard inyecta solo al elegir github. Configs v0.6 sin clave `github` cargan idéntico — verificado via fixture `v0.6-no-github.json`.
4. **D-20 color isolation (Phase 26 CONTEXT.md):** Todos los outputs CLI nuevos consumen `createFormatter(stream)` de `src/cli/format.js`. Invariante v0.5 preservado — `picocolors` solo importado desde `format.js`.
5. **Hybrid client+provider polling path (Phase 25 Open Q #2 RESOLVED):** `startPolling` acepta `client` o `provider`; cuando ambos presentes, prefiere `client.listIssues` por etag/304 optimization. Provider-only path es seed para Phase 27 contract matrix.
6. **Clock injection duck-typed (Phase 25 Open Q #3 RESOLVED):** `{setTimeout, clearTimeout, now}` injectable; mirror dispatcher.js `deps` pattern. Tests usan `createTestClock()` con virtual time advance — wall-time del suite 122ms.
7. **Warn-and-continue retry exhaustion (Phase 25 Open Q #1 RESOLVED):** Tras 3 retries fallidos, polling emite `polling.error{attempt:3}` y continúa al siguiente tick. NO emite `polling.stopped` event (simplicity).
8. **Matrix usa factories directos (Phase 27 SUMMARY):** `createPlaneProvider`/`createGitHubProvider` invocados directamente en el contract matrix en lugar de `getProvider`. Justificación: `getProvider` invoca `loadConfig` fragile en CI. SC#1 ROADMAP preservado — el contract evaluado es del adapter, no del registry wiring.
