<objective>
Cerrar el v0.7 tech debt DAEMON-01: el operador no tenía visibilidad del comportamiento per-tick del daemon de polling. Esta phase añade el flag ortogonal `--verbose` que activa emisión de un evento cross-repo agregado `polling.tick.summary` (D-10) por tick. En foreground (`--no-daemon --verbose`), el evento se renderiza también a stdout via `createFormatter` (D-09) — TTY columnar humano con colores o NDJSON byte-determinístico según TTY-detect.

Purpose: Operador puede diagnosticar el comportamiento del polling sin parsear el NDJSON sink raíz — útil para shakedown post-config y troubleshooting interactive.
Output: Nuevo evento `polling.tick.summary` en la closed taxonomy, helper `pollingTickSummary`, emisión en el tick loop, foreground subscriber con TTY-aware rendering, flag Commander `--verbose`, test integration spawn.

**Cambio adyacente acceptable (declarado explícitamente)**: tras este plan, `runForegroundPolling` construye e inyecta SIEMPRE un `baseLogger` (vía `createLogger({ sessionId: 'polling', minLevel: 'info' })`) al `startPolling` — antes lo llamaba sin logger. Consecuencia: en foreground, los eventos `polling.tick`/`polling.tick.summary`/`polling.dispatch`/`polling.error` ahora SÍ llegan al sink NDJSON raíz (`~/.kodo/logs/<session>.ndjson`). Este es el sink diseñado para telemetría estructurada (D-18 Phase 28 separation of concerns), por lo que el cambio es desiderable, no regresión.
</objective>
