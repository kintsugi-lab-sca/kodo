<objective>
Enrich `GET /status` so every active session carries a read-only `provider_state` + `provider_state_reason`. The cache/dedup/fail-open logic lives in a NEW pure, dependency-injected module `src/server/provider-state.js` (testable without booting the HTTP server), wired into the existing `/status` enrichment map. Register the `provider.state.fetch.failed` event. Update the STATE.md "9-method contract" invariant note.

Purpose: This is the "enrichment" layer of Phase 40 and the payload Phase 43 renders. It closes ROMAN-150 end-to-end on the data side: a Plane "In Review" task surfaces in `/status` even when kodo's local lifecycle says the session ended. `provider_state` is a strictly READ-ONLY lane — never written to `state.json`, never coupled to `alive`/`elapsed_min` (reconcileTick remains the sole writer of `alive`).

Why a new module (not inline in server.js): the server's `/status` HTTP handler has NO existing test harness (no `test/server.test.js`; the server needs network + provider to boot). The cache/dedup/fail-open logic is the riskiest code in this phase. Extracting it as `createProviderStateResolver({ provider, logger, ttlMs, now })` makes it unit-testable with injected mocks (the Nyquist test target) and keeps `server.js` a thin wiring change.

Output: `provider_state` + `provider_state_reason` on each `/status` row; a resolver module + its tests; the new logger event; the doc update.

DESIGN NOTE FOR THE REVIEWER (D-06 — deliberate reinterpretation of PSTATE-04):
PSTATE-04 literally says "omit the field if the call fails or the provider does not support the method". This plan deliberately reinterprets that as **`provider_state = null` WITH a populated `provider_state_reason`** (`'unsupported'` = permanent, `'fetch-failed'` = transient), NOT a silently-omitted field. Reason: Phase 43 success criterion 2 requires distinguishing THREE visual states (ok / unsupported / fetch-failed) "reusing the supported/reason field of Phase 40". Silently omitting the field makes that distinction impossible — the exact lesson v0.9 learned with `listComments`/`supported`. This is intentional, not a bug.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| provider API → resolver | getTaskState issues outbound HTTP to Plane/GitHub with ids derived from the session record; the returned state crosses into the /status response |
| resolver failure → NDJSON log | a getTaskState rejection produces a provider.state.fetch.failed event whose `error` field could carry provider response text |
| /status response → dashboard clients | provider_state / provider_state_reason are new additive fields consumed downstream (Phase 43) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-04 | Information disclosure | provider.state.fetch.failed `error` field | mitigate | providerStateFetchFailed uses an EXPLICIT whitelist {task_id, provider, error} — never `...fields`; the resolver passes `err.message` (a string), never the full error/response object, so tokens/secrets in headers/body never reach the NDJSON (D-15, LOG-12). |
| T-40-05 | Denial of Service | per-poll getTaskState fan-out (N+1) | mitigate | task_id-keyed cache (TTL = PENDING_CACHE_TTL_MS, 30s) + in-flight dedup keep two in-TTL polls at ≤N calls; asserted by the cache + dedup unit tests in Task 2 (D-02/D-03/D-04). |
| T-40-06 | Tampering | provider_state coupling to lifecycle | mitigate | provider_state is read-only: the resolver module structurally cannot write state.json (no such import), and the /status wiring does not touch alive/elapsed_min. reconcileTick stays the sole writer of alive. Asserted by the no-import grep (Task 2) + no-saveState grep (Task 3). |
| T-40-07 | Denial of Service | one provider error 500-ing /status | mitigate | Promise.allSettled (never Promise.all) — a single row's rejection becomes `provider_state_reason:'fetch-failed'` and /status still returns 200 (fail-open per row). |
| T-40-08 | Tampering / SSRF | getTaskState outbound fetch | accept | ids/refs derive from the existing session record and flow through the same provider client used by getTask/listComments (no new URL surface). Inherited boundary, no new mitigation. |
| T-40-SC | Tampering | npm/pip/cargo installs | accept | This plan adds ZERO dependencies. No package-legitimacy gate required. |
</threat_model>
