# Phase 50.1 Plan 02: Repunte del enrich de App.js a la fuente GSD Summary

El enrich client-side de `App.js` ahora lee el progreso vivo del bloque `progress:` del STATE.md del worktree GSD real (`computeRealWorktreePath(project_path, session_id)` → `readGsdProgress`), gateado por `row.gsd === true`, con keep-last-good re-keyed a `session_id`; el display (progCell/deriveAnyProgress/columna prog/cero-color) se reusó intacto y el shim transitorio `readProgress` quedó eliminado.

## What Was Built

- **Import repuntado** (`src/cli/dashboard/App.js:70-72`) — `readProgress` → `readGsdProgress` desde `./progress.js`; nuevo import `computeRealWorktreePath` desde `../../session/state.js`.
- **Bloque enrich reescrito** (`App.js` ~346-368) — la fuente del progreso vivo pasa de `~/.kodo/progress/<task_id>.json` (superficie equivocada) al STATE.md del worktree GSD real:
  - **Gate DG-03:** `if (row?.gsd !== true) return { ...row, progress: { status: 'no-progress' } }` — solo las filas GSD se leen; no-GSD → '—'.
  - **DG-04:** la ruta se deriva con `computeRealWorktreePath(row.project_path, row.session_id)`, NUNCA de `row.worktree_path` persistido (Pitfall 1 — apunta a `.bg-shell` incorrecto).
  - **T-501-traversal:** guard anti-traversal del `session_id` ANTES de construir la ruta (`!sessionId.includes('/')/'\\'/'..'`, String.includes NO regex). Falta `project_path`/`session_id` o no usable → '—'.
  - **DG-07 keep-last-good:** el `progressLastGoodRef` Map ahora se keya por `session_id`; lógica de estados conservada exacta ('ok' → set + return; 'error' → prev presente → N/M, ausente → 'error'; 'no-progress' → return res).
  - Comentario del bloque actualizado para reflejar la fuente STATE.md.
- **Shim `readProgress` eliminado** (`src/cli/dashboard/progress.js`) — el shim transitorio del Plan 01 (devolvía `{no-progress}`) quedó como dead code tras el rewire. `grep -rn readProgress src test` confirmó cero consumidores restantes → eliminado.

## How It Works (load-bearing)

El `...s` que `src/server.js` ya emite en `/status` incluye `gsd`, `project_path` y `session_id`, así que el enrich es enteramente CLIENT-SIDE (DG-06): cero cambios en server.js, cero endpoints, cero deps. El display de Phase 50 (`progCell` 4 estados, `deriveAnyProgress`, columna `prog` con `COLS.prog=7`, cero-color) se reusó INTACTO (DG-05) — el único delta fue de dónde lee el enrich. La re-keying a `session_id` alinea la memoria de last-good con la identidad de la fuente (el STATE.md vive en el worktree de la sesión, no en un artefacto por task_id).
