### `src/cli/dashboard/format.js` (utility, transform) — TUI-10 (fix mecánico)

**Analog:** `stateBadge` (`format.js:136-138`) — el patrón `state ?? status` YA establecido.

**Patrón de referencia `stateBadge`** (`format.js:121-138`):
```javascript
export const STATE_BADGES = Object.freeze({
  running: { glyph: '▶', color: 'green', label: 'running' },
  idle: { glyph: '⏸', color: 'yellow', label: 'idle' },
  'needs-input': { glyph: '🔔', color: 'cyan', label: 'needs-input' },
  dead: { glyph: '✗', color: 'red', label: 'dead' },
});

export function stateBadge(state) {
  return STATE_BADGES[state ?? ''] ?? {};
}
```

**Código a modificar `statusColor`** (`format.js:91-98`):
```javascript
export function statusColor(status, alive) {
  if (status === 'running' && !alive) return { color: 'red' };
  if (status === 'running') return { color: 'green' };
  if (status === 'review') return { color: 'cyan' };
  if (status === 'error') return { color: 'magenta' };
  if (status === 'done') return { dim: true };
  return {};
}
```

**Problema (evidencia audit):** `statusColor` solo ve `session.status` (v2). Sesiones v3 con `status` v2 `null` (idle / needs-input) → celda `status` sin color. `rowCells` (175) ya llama `statusLabel(session.status ?? '', session.alive)` — solo el badge izquierdo (vía `stateBadge`) colorea.

**Patrón a aplicar (a discreción del planner, D-08 LOCKED):** que `statusColor` derive también del estado v3 (`session.state`), igual que `stateBadge`. El planner elige la firma exacta. Dos opciones coherentes con la base:
  - **Opción A (alineada con `stateBadge`):** `statusColor(status, alive, state)` → si `status` no matchea ninguna rama v2 y existe un color en `STATE_BADGES[state]`, devolver `{ color: STATE_BADGES[state].color }`. Reutiliza la paleta YA LOCKED (idle=yellow, needs-input=cyan, dead=red) sin duplicar nombres de color.
  - **Opción B:** pasar el `Session` completo y aplicar internamente `state ?? status` antes del switch.

**Invariantes LOCKED a preservar (D-08 + byte-determinismo):**
  - Devolver NOMBRES de color ink (`'green'|'red'|'cyan'|'magenta'|'yellow'`) o `{ dim: true }`, JAMÁS ANSI (ink convierte el nombre internamente).
  - Reusar la paleta existente: NO inventar colores nuevos. `dead`/zombie = `red`, `idle` = `yellow`, `needs-input` = `cyan` (ya en `STATE_BADGES`).
  - `red` reservado al estado muerto/zombie (comentario `format.js:17`); `magenta` para `error` (distinto del red).
  - Byte-determinismo de `--json` / `NO_COLOR`: la función es pura, sin I/O. El call-site `rowCells` (170-178) debe seguir pasando el campo correcto; verificar que `SessionTable.js` (que llama `statusColor` vía el import de `format.js:25`) reciba el argumento extra si la firma cambia.

**Call-sites de `statusColor` a actualizar si cambia la firma:** `grep -rn "statusColor" src/cli/dashboard/`. Hoy se importa en `SessionTable.js:25` y se usa en el render de la celda `status`. Si se añade `state` como tercer arg, propagarlo desde la fila (`row.state`).
