# Phase 39.1 Plan 04: Overlay de comentarios — distinguir 'unsupported' de 'empty' Summary

TUI-15 cerrado: el server señala `supported:false` con un campo ADITIVO en `/comments/{taskId}` cuando el provider no implementa `listComments`, y la TUI (overlay ink + vista HTML web) renderiza un mensaje distinto de "no comments yet" — honestidad de UI entre estado permanente (provider sin soporte) y transitorio (tarea sin comentarios aún).

## What Was Built

- **Task 1 (server.js):** El handler `GET /comments/{taskId}` calcula `const supported = typeof provider.listComments === 'function'` y serializa `{ comments, supported }`. Cuando `supported`, llama `await provider.listComments(...)`; cuando no, `comments = []`. El 404 'Session not found' y el 500 catch quedan intactos. Cambio byte-compatible: clientes viejos ignoran `supported` (invariante v0.9 "respuestas JSON aditivas").
- **Task 2 (client.js):** El `@typedef FetchCommentsResult` extendido a `data: { comments: any[], supported?: boolean }`. El runtime NO cambia: `fetchComments` ya hace pass-through de `data` entero tras la guard `Array.isArray(data.comments)` — `supported` viaja sin lógica adicional. Solo typedef + JSDoc.
- **Task 3 (App.js + SessionTable.js + test):**
  - `App.js`: constante exportada `OVERLAY_COMMENTS_UNSUPPORTED = 'comments not supported by this provider'` (literal-estable, redundancia textual legible bajo NO_COLOR). En el handler `c`, dentro de `if (res.ok)`, `res.data.supported === false` mapea a `status = 'unsupported'` ANTES de la lógica ok/empty (envuelta en `else`).
  - `SessionTable.js`: import de `OVERLAY_COMMENTS_UNSUPPORTED` + rama `else if (snap.status === 'unsupported')` en `renderOverlay` que asigna el copy con `dimColor` (NO rojo — es informativo, no un error).
  - `test/dashboard-table.test.js`: describe `TUI-15` con 3 tests — (a) `OVERLAY_COMMENTS_UNSUPPORTED` byte-estable y distinto de `OVERLAY_COMMENTS_EMPTY`, (b) `c` con `supported:false` pinta el copy unsupported y NO el empty, (c) `c` con `supported:true` + vacíos sigue pintando empty (sin regresión).

## Deviations from Plan

### Decisión D-09: vista HTML web — APLICADA (no diferida)

El plan dejaba a criterio defensivo si actualizar la vista HTML web (`toggleComments`, server.js:167-195) o documentarla como deuda acotada. **Se aplicó** el mismo trato: cuando `data.supported === false`, la caja de comentarios HTML muestra "Comentarios no soportados por este provider" en lugar de "Sin comentarios". Rationale: es un cambio aditivo de 3 líneas, consistente con el overlay ink, y cierra D-09 por completo sin riesgo (la rama `supported === false` es nueva, no toca el path vacío/lleno existente). Se incluyó en el commit de Task 3 (mismo `src/server.js`).

### Tests: ubicación

El plan especificaba añadir el test a `test/dashboard-table.test.js` (y la acceptance gateaba ahí). Se respetó: el harness fake-clock + `injectProps`/`drain`/`okResponse`/`FIXTURE` de ese archivo soporta el render completo de `App` con un `fetchFn` que enruta `/status` vs `/comments/`, idéntico al patrón de `test/dashboard-overlay.test.js`. No se duplicó cobertura en `dashboard-overlay.test.js`.

### Task 1 TDD — server test hermético

El plan permitía documentar y diferir la verificación E2E al overlay (Task 3) si no existía infra de server-test simple. **No existe** un harness hermético para el handler `/comments` aislado (el handler vive inline dentro de `createServer` en `startServer`, que requiere registry init + config + HTTP real). La verificación del cambio de shape del server se cubre vía: (1) grep acceptance byte-level + `node --check`, y (2) el test E2E de Task 3 que ejerce `supported:false` y `supported:true` a través del path real client→App→renderOverlay contra la shape exacta del server. RED/GREEN se materializó como la suite de Task 3 (roja sin la constante + el mapeo, verde con ellos).
