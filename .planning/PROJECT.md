# kodo

## What This Is

kodo es un bridge entre sistemas de gestiГіn de tareas y sesiones de Claude Code via cmux. La arquitectura es provider-agnostic: cualquier sistema de tareas (Plane, GitHub Issues, ClickUp, local) se integra implementando la interfaz `TaskProvider` de 9 mГ©todos. Plane CE es el primer adaptador implementado y validado. Desde v0.3, kodo orquesta sesiones Claude bajo el workflow GSD (1 tarea Plane = 1 fase GSD) con bootstrap automГЎtico, gate de verificaciГіn contra `VERIFICATION.md`, y observabilidad NDJSON end-to-end. Desde v0.4, una segunda label `kodo:gsd-quick` arranca sesiones one-shot sin plan/execute/verify, con el mismo lock + skip-permissions y orchestrator que las distingue.

## Core Value

Cualquier sistema de tareas puede ser el motor de kodo вЂ” cambiar de proveedor no requiere reescribir la lГіgica de sesiones, health checks ni orquestaciГіn. El mismo sistema dispara dos modos GSD: full (`kodo:gsd`, multi-fase con verify) y quick (`kodo:gsd-quick`, one-shot), sin acoplar el cГіdigo GSD al proveedor.

## Milestone v0.14 ConfiguraciГіn editable desde el dashboard (shipped 2026-06-30, histГіrico)

**Target features:**
- **Editor de proyectos** вЂ” lista los proyectos del provider en vivo (`listProjects()` Plane/GitHub) в†’ eliges uno в†’ mapeas/editas/quitas su ruta local (+ mГіdulos opcionales), escribiendo a `~/.kodo/projects.json`. - **Editor de ajustes comunes** вЂ” `claude.default_model` + `claude.max_parallel`, estados del provider (`states.trigger`/`review`/`done`), thresholds del server (`server.idle_threshold_min`/`stuck_threshold_min`) y colores cmux (`cmux.colors`), escribiendo a `~/.kodo/config.json`.
- **Escritura local, cero endpoints nuevos** вЂ” el dashboard persiste vГ­a filesystem local / shell-out (`saveConfig`/`saveProjects` o `kodo config`), nunca un `POST` nuevo en `src/server.js`. Apoyado en el precedente de v0.13 (la TUI ya escribe shelleando `kodo adopt` vГ­a `execFile`).
- **Aviso de reinicio (sin hot-reload)** вЂ” tras guardar, el dashboard indica que hay que reiniciar server/daemon para aplicar; honesto y simple, coherente con el comportamiento actual (la config se carga en memoria al arrancar).
- **Secrets intactos** вЂ” las API keys nunca se editan ni se muestran; siguen viviendo exclusivamente en `~/.kodo/.env` (invariante de seguridad).

**Key context:**
- **2ВЄ ruptura consciente de "TUI read-only".** El dismiss de sesiones dead (v0.10) fue la 1ВЄ; este editor es la 2ВЄ. La config NO vive en el server (vive en `~/.kodo/config.json` + `projects.json`), asГ­ que el dashboard la escribe localmente вЂ” preserva "cero endpoints nuevos desde v0.10".

Capturar rutas/valores requiere un componente de text-input (controlado vГ­a `useInput` o dep tipo `ink-text-input`) вЂ” candidato a research/decisiГіn de diseГ±o.
- **Listar proyectos requiere conexiГіn** al provider (`listProjects()` en vivo); el editor debe degradar con gracia si la conexiГіn falla (TUI never-throws sigue en pie).
- **VehГ­culo: solo el dashboard.** El wizard `kodo config` lineal se deja como estГЎ en este milestone.

## Constraints
